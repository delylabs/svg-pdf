# Supported features

## Supported

Shapes (`path`/`rect`/`circle`/`ellipse`/`line`/`polygon`/`polyline`), nested `<g>` transforms, `<use>`/`<defs>`/`<symbol>`, nested `<svg>` (own `x`/`y`/`width`/`height`/`viewBox`, clipped to its viewport by default — see below), `<symbol>` (clipped to its resolved `width`/`height` by default, same as nested `<svg>` — see below), `preserveAspectRatio` (`meet`/`slice`, all 9 alignment keywords) on the root `<svg>`, `<symbol>`, and nested `<svg>` alike, linear/radial gradients, `<pattern>` tiling (translate/scale placement only — see below), `<marker>` (`marker-start`/`-mid`/`-end`, `orient="auto"`/`"auto-start-reverse"`/fixed angle, `markerUnits`, `refX`/`refY`, `viewBox`, `overflow="visible"` — see below), `<clipPath>` (including `objectBoundingBox` units), `stroke-dasharray`, `mix-blend-mode`, `<style>` selectors (full CSS selector support — combinators, pseudo-classes, attribute selectors — via the `css-select` library, not just tag/class/id — see below), `<text>`/`<tspan>` with chunk-wide `text-anchor`, `text-transform`, `letter-spacing`/`word-spacing`, `xml:space="preserve"`/`white-space: pre`, and `<textPath>` (see below) (standard-14 fonts by default; a custom font is embedded instead if the SVG declares a matching `@font-face { src: url(data:...) }`, or if a `fetchFont` function is passed in and returns bytes for the requested `font-family`/`weight`/`style` — see [`usage.md`](usage.md)), `<image>` (inline `data:` URIs always; external `http`/`https` URLs only if a `fetchImage` function is passed in — see [`usage.md`](usage.md) — since fetching them by default would be an SSRF risk for anyone converting untrusted, user-supplied SVGs), and `<a href>` as a clickable PDF link annotation (see below).

`<pattern>` support has two scope limits, both from `@libpdf/core` having no way to position a tiling pattern other than through absolute, axis-aligned numbers: a pattern reached through a rotated/skewed transform (its own `patternTransform` or an ancestor `<g>`) is skipped with a warning rather than tiled incorrectly, and pattern _content_ is limited to shapes with a solid fill/stroke (no nested gradients/patterns, `<text>`, or `<image>` — each skipped individually with a warning, same fail-safe policy as everywhere else). `<marker>` content has the same solid-fill/stroke-only limit (both are built from a `@libpdf/core` resource with no way to register a font/image/shading/nested pattern into it), but marker _placement_ itself has no such restriction — a marker is painted as an ordinary positioned object, so it rotates/scales freely. A `<marker>`'s content defaults to clipped to its `markerWidth`/`markerHeight` viewport, per spec; `overflow="visible"` opts out of that clip (a common technique for an arrowhead whose path coordinates intentionally extend past a small nominal viewport) — since a PDF Form XObject's own `/BBox` is always a hard clip with no "don't clip" equivalent, this is handled by growing the `/BBox` to fit the marker's actual shape content instead of the nominal viewport size. `fill-opacity`/`stroke-opacity`/element `opacity` on `<pattern>`/`<marker>` _content_ (as opposed to on the shape the pattern/marker is applied to, which is fully supported) is also not honored — both are built from a bare `operators: Operator[]` array with no resource dictionary of its own, so there's nowhere to register the `ExtGState` PDF needs for partial opacity; the content is drawn fully opaque instead, with a warning, rather than approximated by pre-blending the color against an assumed backdrop (which would only be correct over a solid, known background).

A `@font-face` rule is only read automatically if its `src` includes a `url(data:...)` — an externally-hosted `src` is skipped with a warning (same reasoning as `<image>`'s external-URL opt-in: no network request is ever made without the caller providing one, here via `fetchFont`). Matching a `<text>`'s `font-family`/`font-weight`/`font-style` against a `@font-face` is a literal, case-insensitive string comparison, not full CSS font-matching (no weight ranges, no `font-stretch`).

`dx`/`dy`/`rotate` on `<text>`/`<tspan>` accept the full per-character list form, not just a single scalar shift — a run with 2+ `dx`/`dy` values or any `rotate` at all is drawn one character at a time (missing list entries default to 0 for `dx`/`dy`; `rotate` repeats its last value for any character past its end, per spec).

`letter-spacing`/`word-spacing` are drawn with PDF's own character/word-spacing text-state operators rather than by hand-splitting runs into characters/words — `word-spacing` in particular is a PDF-spec quirk: it only affects single-byte space codes, which is how a standard-14 font is always encoded, but an embedded custom font (via `fetchFont`/`@font-face`) is typically encoded as a multi-byte (CID-keyed) font internally, for which PDF readers don't apply word-spacing at all — so `word-spacing` on custom-font text may have no visible effect, a limitation of the PDF format itself rather than of svg-pdf's implementation. `letter-spacing` isn't affected by this and works with either font source. `text-transform` only performs simple case mapping (no locale-aware casing).

An `<a href>`'s clickable region is the axis-aligned bounding box of whatever shapes/text/images it wraps (including invisible `fill="none"` shapes — a common "transparent clickable overlay" pattern), not the exact outline of that content — an `<a>` wrapping a star icon gets a rectangular click target around the star, same as most SVG-to-PDF converters. `href="#fragment"` (an internal same-page anchor) is skipped with a warning: each SVG becomes one standalone PDF page, with no multi-page document to navigate within, so there's no equivalent destination. Any other `href` (external URL, `mailto:`, `tel:`, etc.) is written through as-is, unvalidated — unlike `<image>`'s external-URL fetching, writing a link annotation never makes a network request, so there's no SSRF-style opt-in needed here.

`<textPath href="#id">` draws its text along the referenced `<path>`'s geometry, one character at a time (each with its own position and rotation matching the path's tangent there) rather than one flat string — the referenced element's `d` is flattened into a polyline (straight/Bézier commands exactly; elliptical arcs are converted to Béziers first, same as everywhere else path data is measured in this codebase, so they're supported too, not approximated). `startOffset` (a plain number or a `%`) and the referenced path's own `pathLength` (which rescales what its length units mean) are both honored; `text-anchor` shifts the starting position along the path the same way it shifts along an axis for regular text. Not supported: `textLength`/`lengthAdjust` (stretching/compressing text to fit an exact length) and `<tspan>` children nested inside a `<textPath>` (only its own direct text content is drawn) — both skipped with a warning rather than silently ignored. A `<textPath>` also doesn't participate in `fetchFont`/`@font-face` custom-font matching yet — it always draws with the standard-14 fallback font.

A nested `<svg>`'s own `x`/`y`/`width`/`height` may be percentages, resolved against the parent viewport (its viewBox size, if it has one, else its raw width/height); `width`/`height` default to 100% of the parent when absent, per spec, rather than requiring explicit numbers. A `<symbol>` (rendered through `<use>`) gets the same treatment: its resolved viewport comes from `<use>`'s own `width`/`height` if present, else the `<symbol>`'s own `width`/`height`, else 100% of the current viewport — and `<use>`'s own `x`/`y` may be percentages too. Both a nested `<svg>` and a `<symbol>` clip their content to that resolved viewport by default, per spec; `overflow="visible"` on either opts out of the clip.

A shape's own `%`-valued geometry (`x`/`y`/`width`/`height`/`rx`/`ry` on `<rect>`, `cx`/`cy`/`r` on `<circle>`, `cx`/`cy`/`rx`/`ry` on `<ellipse>`, `x1`/`y1`/`x2`/`y2` on `<line>`) resolves against the current viewport's size — horizontal values against its width, vertical against its height, and `r` (not purely one or the other) against the viewport diagonal, same convention CSS/SVG use elsewhere. A shape inside a nested `<svg>` or a `<symbol>` correctly resolves its percentages against that local viewport's own size, not the root's.

`<style>` selector matching is delegated to the `css-select` library (via a small adapter over this codebase's `@xmldom/xmldom`-based element tree) rather than hand-rolled — combinators (descendant/child/sibling), pseudo-classes (`:not()`, `:first-child`, ...), and attribute selectors (`[fill="none"]`, ...) are all supported, matched with real CSS specificity (id > class/attribute/pseudo-class > type/pseudo-element), ties broken by source order. A selector that fails to parse (genuinely invalid CSS syntax) is skipped with a warning, same fail-safe policy as everywhere else — the rest of the rule's selector list (if comma-separated) and all other rules still apply.

## Not yet supported

- `<mask>` — also currently blocked by `@libpdf/core` itself (no soft-mask option in its `ExtGState` API)
- `filter="url(#...)"` (blur/drop-shadow) — a genuine PDF vector limitation, not a scope choice
