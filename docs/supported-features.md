# Supported features

## Supported

Shapes (`path`/`rect`/`circle`/`ellipse`/`line`/`polygon`/`polyline`), nested `<g>` transforms, `<use>`/`<defs>`/`<symbol>`, nested `<svg>` (own `x`/`y`/`width`/`height`/`viewBox`, clipped to its viewport by default — see below), `preserveAspectRatio` (`meet`/`slice`, all 9 alignment keywords) on the root `<svg>`, `<symbol>`, and nested `<svg>` alike, linear/radial gradients, `<pattern>` tiling (translate/scale placement only — see below), `<marker>` (`marker-start`/`-mid`/`-end`, `orient="auto"`/`"auto-start-reverse"`/fixed angle, `markerUnits`, `refX`/`refY`, `viewBox` — see below), `<clipPath>` (including `objectBoundingBox` units), `stroke-dasharray`, `mix-blend-mode`, `<style>` selectors (tag/class/id, and no-combinator compound combinations of those, e.g. `rect.big`, `.a.b`), `<text>`/`<tspan>` with chunk-wide `text-anchor`, `text-transform`, `letter-spacing`/`word-spacing`, `xml:space="preserve"`/`white-space: pre`, and `<textPath>` (see below) (standard-14 fonts by default; a custom font is embedded instead if the SVG declares a matching `@font-face { src: url(data:...) }`, or if a `fetchFont` function is passed in and returns bytes for the requested `font-family`/`weight`/`style` — see [`usage.md`](usage.md)), `<image>` (inline `data:` URIs always; external `http`/`https` URLs only if a `fetchImage` function is passed in — see [`usage.md`](usage.md) — since fetching them by default would be an SSRF risk for anyone converting untrusted, user-supplied SVGs), and `<a href>` as a clickable PDF link annotation (see below).

`<pattern>` support has two scope limits, both from `@libpdf/core` having no way to position a tiling pattern other than through absolute, axis-aligned numbers: a pattern reached through a rotated/skewed transform (its own `patternTransform` or an ancestor `<g>`) is skipped with a warning rather than tiled incorrectly, and pattern _content_ is limited to shapes with a solid fill/stroke (no nested gradients/patterns, `<text>`, or `<image>` — each skipped individually with a warning, same fail-safe policy as everywhere else). `<marker>` content has the same solid-fill/stroke-only limit (both are built from a `@libpdf/core` resource with no way to register a font/image/shading/nested pattern into it), but marker _placement_ itself has no such restriction — a marker is painted as an ordinary positioned object, so it rotates/scales freely.

A `@font-face` rule is only read automatically if its `src` includes a `url(data:...)` — an externally-hosted `src` is skipped with a warning (same reasoning as `<image>`'s external-URL opt-in: no network request is ever made without the caller providing one, here via `fetchFont`). Matching a `<text>`'s `font-family`/`font-weight`/`font-style` against a `@font-face` is a literal, case-insensitive string comparison, not full CSS font-matching (no weight ranges, no `font-stretch`).

`letter-spacing`/`word-spacing` are drawn with PDF's own character/word-spacing text-state operators rather than by hand-splitting runs into characters/words — `word-spacing` in particular is a PDF-spec quirk: it only affects single-byte space codes, which is how a standard-14 font is always encoded, but an embedded custom font (via `fetchFont`/`@font-face`) is typically encoded as a multi-byte (CID-keyed) font internally, for which PDF readers don't apply word-spacing at all — so `word-spacing` on custom-font text may have no visible effect, a limitation of the PDF format itself rather than of Plotify's implementation. `letter-spacing` isn't affected by this and works with either font source. `text-transform` only performs simple case mapping (no locale-aware casing).

An `<a href>`'s clickable region is the axis-aligned bounding box of whatever shapes/text/images it wraps (including invisible `fill="none"` shapes — a common "transparent clickable overlay" pattern), not the exact outline of that content — an `<a>` wrapping a star icon gets a rectangular click target around the star, same as most SVG-to-PDF converters. `href="#fragment"` (an internal same-page anchor) is skipped with a warning: each SVG becomes one standalone PDF page, with no multi-page document to navigate within, so there's no equivalent destination. Any other `href` (external URL, `mailto:`, `tel:`, etc.) is written through as-is, unvalidated — unlike `<image>`'s external-URL fetching, writing a link annotation never makes a network request, so there's no SSRF-style opt-in needed here.

`<textPath href="#id">` draws its text along the referenced `<path>`'s geometry, one character at a time (each with its own position and rotation matching the path's tangent there) rather than one flat string — the referenced element's `d` is flattened into a polyline (straight/Bézier commands exactly; elliptical arcs are converted to Béziers first, same as everywhere else path data is measured in this codebase, so they're supported too, not approximated). `startOffset` (a plain number or a `%`) and the referenced path's own `pathLength` (which rescales what its length units mean) are both honored; `text-anchor` shifts the starting position along the path the same way it shifts along an axis for regular text. Not supported: `textLength`/`lengthAdjust` (stretching/compressing text to fit an exact length) and `<tspan>` children nested inside a `<textPath>` (only its own direct text content is drawn) — both skipped with a warning rather than silently ignored. A `<textPath>` also doesn't participate in `fetchFont`/`@font-face` custom-font matching yet — it always draws with the standard-14 fallback font.

A nested `<svg>` needs an explicit numeric `width`/`height` — per spec these default to 100% of the parent viewport when absent, but resolving that requires tracking a live "current viewport size" that nothing else in the codebase does yet (percentage units are only resolved once, for the root `<svg>`'s own size), so a nested `<svg>` missing them is skipped with a warning rather than guessed at.

## Not yet supported

- CSS selectors beyond tag/class/id/compound combinations (no combinators, pseudo-classes, or attribute selectors)
- Per-character `dx`/`dy`/`rotate` arrays on `<tspan>` (only scalar `dx`/`dy` are read; chunk-wide `text-anchor` is measurement-based, but positioning within a chunk is still a single-line flow heuristic, not per-character)
- `<mask>` — also currently blocked by `@libpdf/core` itself (no soft-mask option in its `ExtGState` API)
- `filter="url(#...)"` (blur/drop-shadow) — a genuine PDF vector limitation, not a scope choice
- Percentage units on anything other than the root `<svg>`'s own `width`/`height` (e.g. `x="50%"` on a nested `<svg>`, `width="100%"` on a shape) — resolving these generally requires threading a live "current viewport size" through parsing, which nothing in the codebase does yet
